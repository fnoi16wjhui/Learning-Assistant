# 信息获取 / 数据采集基础架构

## 目标与边界

本模块负责 Course Agent 的信息获取与数据采集：从网络学堂、清华邮箱、教务/课表等异构来源拉取原始信息，清洗并归一化为下游可消费的结构化课程任务。

整体数据流如下：

```text
Data Sources -> Adapters -> Parsers -> Models -> Pipeline -> Storage/Logs -> Downstream
```

核心设计原则是让系统承担复杂性。登录状态、Cookie、IMAP UID、HTML/MIME 噪音、增量去重、落库和日志都必须被封装在对应层内，顶层调度只面对稳定、统一、可验证的接口。

## 分层架构

### Adapters：原始数据接入层

职责：

- 维护网络会话、Cookie、IMAP 连接、超时、重试和鉴权入口。
- 从网络学堂、邮箱、教务系统等来源拉取原始 HTML、JSON、MIME、ICS 或字节流。
- 为每条原始记录附带 `source`、`raw_id`、`content_type`、`fetched_at` 和最小 metadata。

禁止：

- 禁止清洗 HTML、解析正文、提取 DDL 或判断任务类型。
- 禁止构造 `CourseTask` 或 `ScheduleItem`。
- 禁止硬编码学号、密码、邮箱授权码、Cookie、Token 或 API Key。
- 禁止吞掉登录失败、超时、鉴权失效等异常；错误必须带来源和阶段上下文。

这样分层的原因：Adapters 面对最不稳定的外部系统。把网络与鉴权封装在此层，后续替换登录策略、请求库或数据源时，不会污染解析逻辑和数据模型。

### Parsers：纯解析与清洗层

职责：

- 接收 Adapters 产出的原始字符串或字节流。
- 去除 HTML 标签、邮件签名、页脚、样式噪音和 MIME 包装。
- 提取标题、正文、附件链接、显式时间字段和 DDL。
- 输出通过 Pydantic 校验的 `CourseTask`。

禁止：

- 禁止发起任何网络请求。
- 禁止读取 `.env`、Cookie、账号密码或远端配置。
- 禁止写 SQLite、日志文件或调用下游 API。
- 禁止做跨来源调度与去重。

这样分层的原因：Parsers 应该是可重复、可离线测试的纯计算模块。同一份原始样本必须稳定得到同一组字段，这样才能用 fixture 做回归验证。

### Models：统一数据契约层

职责：

- 使用 Pydantic v2 定义统一的 `Attachment`、`CourseTask`、`ScheduleItem` 等数据模型。
- 校验必填字段、枚举值、时间类型、附件结构和来源标识。
- 屏蔽网络学堂、邮箱、教务系统之间的字段差异。

禁止：

- 禁止包含网络请求、数据库连接或文件写入。
- 禁止为了适配脏数据而放宽核心字段到不可控的 `dict`。
- 禁止读取敏感环境变量。

这样分层的原因：Models 是下游“内容理解 / Agent 工作流 / 产品交互”稳定依赖的契约。所有来源必须收敛到同一结构，避免下游处理多套不兼容输入。

### Pipeline：调度、校验、去重与交付层

职责：

- 编排 Adapter 和 Parser 的调用顺序。
- 基于 `source + raw_id` 计算 SHA-256 fingerprint。
- 使用 SQLite 进行增量去重。
- 将新增 `CourseTask` 写入 JSONL、SQLite 或未来的下游 Agent API。
- 记录同步结果、异常上下文和去重数量。

禁止：

- 禁止直接维护登录细节或拼接复杂请求。
- 禁止直接解析 HTML/MIME 正文。
- 禁止绕过模型校验向下游交付。
- 禁止把敏感凭证写入 SQLite、JSON、日志或异常消息。

这样分层的原因：Pipeline 是任务流的交通枢纽，关注幂等、去重、持久化和交付，而不承担具体网络协议或文本解析细节。

## Storage / Logs

默认存储使用本地 SQLite，记录 fingerprint、来源、原始 ID、记录类型和发现时间。JSON Lines 用于调试、回放和下游试接入。原始 HTML/MIME 默认不保存，只有在脱敏且明确需要调试时才开启。

日志默认写入 `logs/collector.log`，记录任务开始、渠道结果、去重数量和异常上下文。日志不得出现密码、授权码、完整 Cookie、Token 或 API Key。

## main 调度

`main.py` 是保守入口：默认不访问网络。只有显式指定 channel 且传入 `--allow-network`，未来才会进入真实同步流程。这样可以保证新同学运行项目时先走 harness，而不是意外访问真实校内系统。

调度原则：

- 邮件轮询默认 15 分钟，网络学堂默认 30 分钟，教务默认 120 分钟。
- 单一渠道失败不应导致其他渠道停止。
- 网络失败使用有限重试和指数退避。
- 每轮同步输出结构化摘要，便于下游和人工排查。

## 下游交付

下游只接收通过 Pydantic 校验且未重复的结构化对象。初期交付方式建议：

- SQLite：作为本地共享状态。
- JSON Lines：用于调试、回放和跨语言消费。
- Agent API 回调：后续接入内容理解与提醒工作流。

下游不应再解析 HTML、MIME 或处理登录状态。

## 子代理 harness

多代理协作时，每个子代理必须按 Plan / Implementation / Reflection 三个检查点推进，避免跨层污染和文件冲突。

### Plan：开工前

子代理需要确认：

- 本次任务属于哪一层：Adapters、Parsers、Models、Pipeline、配置或文档。
- 将创建或修改哪些文件，是否完全在自己的写入范围内。
- 输入是什么、输出是什么，是否跨过上游或下游边界。
- 是否涉及凭证、Cookie、邮箱授权码或 API Key。
- 是否需要 fixture 或 mock 来验证，而不是访问真实账号。

通过标准：

- 文件范围清晰，不覆盖其他代理职责。
- 设计符合四层流水线。
- 敏感信息只来自环境变量或占位模板。
- 验证方式可以独立执行。

### Implementation：提交时

子代理需要验证：

- Adapter 没有解析正文或构造业务模型。
- Parser 没有网络请求、数据库写入或环境变量读取。
- Model 使用 Pydantic v2，字段含义稳定且类型明确。
- Pipeline 先模型校验，再去重，再交付。
- `.env.example` 只包含占位变量。
- 日志输出不会泄露密码、Token、Cookie 或授权码。

通过标准：

- 单次改动高内聚，可以独立 review。
- 错误信息包含渠道和阶段上下文。
- 没有 `try-except pass` 式吞错。
- 新增配置有保守默认值和清晰命名。

### Reflection：交付后

子代理需要自查：

- 是否存在硬编码敏感信息。
- 是否存在跨层污染。
- 是否破坏 `CourseTask` 契约。
- 是否可能重复推送同一来源记录。
- 是否对失败路径有日志和恢复策略。
- 是否触碰了其他代理负责的文件。

通过标准：

- 能说明改动如何进入四层流水线。
- 能说明失败时如何定位问题。
- 能说明本次未完成事项和后续风险。
- 能给出可复现的验收 harness。
